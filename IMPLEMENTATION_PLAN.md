# План выполнения тестового задания (Flutter Kanban, KPI-DRIVE)

## Цель
Сделать **красивую и удобную** канбан-доску “как Trello” (или лучше) на Flutter:
- **Колонки** = папки по `parent_id`
- **Карточки** = задачи (поля: `indicator_to_mo_id`, `parent_id`, `name`, `order`)
- **Drag & Drop**: перенос между колонками (меняем `parent_id`) и сортировка внутри колонки (меняем `order`)
- **Сохранение** изменений на сервер

Приоритет оценки: **визуал + UX + функциональность**, при этом архитектуру держим чистой.

## Публичная “демо-цель” (то, что будет видно директору)
- **Мгновенно понятная доска**: человеческие названия вместо “id папок” (если API не дает названия папок — используем понятные заголовки, например “Без папки / Inbox” + группировка, и показываем “Папка #123” только вторично).
- **Минимум кликов**: перетащи — и готово; тап по карточке — быстрый просмотр; вторичные действия в bottom sheet.
- **Визуальные фишки**:
  - аккуратные анимации перемещения карточек
  - красивый drag preview (карточка “поднимается”, тень/скейл)
  - skeleton/shimmer при загрузке
  - empty states (пустые колонки с понятным текстом)
  - toast/snackbar с **Undo** после перемещения
  - “Saving…” индикатор на карточке при отправке
  - приятные micro-interactions (haptics, hover на desktop/web если включим)
- **Защита от дурака**:
  - запретить перетаскивание во время сохранения конкретной карточки (или ставить в очередь)
  - обработка ошибок API с мягким откатом UI и понятным сообщением
  - отсутствие “сломанных” состояний при плохом интернете

## Архитектура (чтобы не развалилось при доработках)
Выбираем простую, но чистую схему слоев:
- **Data**: DTO, API клиент, мапперы, репозиторий.
- **Domain**: сущности `TaskItem`, `BoardColumn`, use-cases (fetch, move, reorder, save).
- **Presentation**: state management + UI.

State management: `Riverpod` (предпочтительно) или `Bloc/Cubit` — выберем один и не смешиваем.

## План работ (по шагам)
### Шаг 0. Базовая подготовка репозитория
- Создать Flutter проект (если его еще нет) + структура папок под слои.
- Подключить зависимости:
  - сеть: `dio`
  - state: `flutter_riverpod` (или `flutter_bloc`)
  - drag & drop/board: выбрать библиотеку или кастом
  - UI: `google_fonts`, `flutter_svg`, `shimmer` (опционально)
- Настроить темы: светлая/тёмная (если успеваем) в стиле KPI-DRIVE.

### Шаг 1. Реализация API слоя
- `ApiClient` на `dio`:
  - заголовок Authorization (bearer)
  - отправка `multipart/form-data` (form-data)
  - логирование в debug (не в релиз)
- `IndicatorsRepository`:
  - `fetchTasks(...)` возвращает список задач (DTO → Domain)
  - `saveTaskField(...)` для сохранения одного поля
  - (если требуется 2 поля) утилита `saveTaskMoveAndOrder(...)` которая делает 2 запроса или один за другим (в зависимости от фактического API поведения)

### Шаг 2. Модель доски и группировка в колонки
- Преобразовать список задач в структуру:
  - колонка = `parent_id` (ключ)
  - карточки сортируются по `order`
- Обработать кейсы:
  - `parent_id == null` (или 0) — колонка “Inbox/Без папки”
  - дубли/пропуски `order` — нормализация на клиенте для UI (не ломая серверную логику)

### Шаг 3. Экран канбан-доски (визуальный “вау”)
- Layout как в Trello:
  - горизонтальный скролл колонок
  - заголовок колонки + счетчик задач
  - список карточек, скролл внутри
- Карточка задачи:
  - название (`name`) + небольшой subtitle с вторичной инфой (например `#indicator_to_mo_id`)
  - статусы: idle / dragging / saving / error
- Состояния:
  - shimmer при загрузке
  - empty state (когда задач нет)
  - error state с Retry

### Шаг 4. Drag & Drop: перемещение и сортировка
- Реализовать DnD:
  - перетаскивание карточки между колонками → меняем `parent_id`
  - перетаскивание внутри колонки → меняем `order`
  - live preview + анимации перестановки
- Правила UX:
  - во время drag подсвечиваем target колонку
  - автоскролл при поднесении к краям
  - haptic feedback (мобилка)

### Шаг 5. Сохранение на сервер + оптимистичный UI
- При drop:
  - **сразу обновить UI** (optimistic update)
  - отправить сохранение `parent_id` и `order`
  - на успех: снять “saving”
  - на ошибку: показать snackbar + откатить позицию (или предложить Retry)
- Добавить **Undo**:
  - после перемещения показываем snackbar “Перемещено” + Undo
  - Undo делает локальный откат и повторный save
- Защита:
  - очередь/лок на карточку во время сохранения (чтобы не словить гонки)
  - дедуп запросов при быстрых перемещениях (последнее действие важнее)

### Шаг 6. “Прикольные плюшки” (если успеваем — в порядке важности)
- Быстрый поиск по `name` (фильтр на доске).
- Sticky header у колонки при вертикальном скролле списка карточек.
- Переключатель “Compact / Comfortable” плотности карточек.
- Небольшая “статистика” в app bar: всего задач / колонок.
- Тёмная тема (если в KPI-DRIVE уместно).

### Шаг 7. Полировка и надежность
- Обработка всех ответов сервера:
  - валидировать что поля пришли
  - аккуратно переживать неожиданные типы/пустые значения
- Лоадеры:
  - глобальный (первый вход)
  - локальный на карточке (saving)
- Текст ошибок человеческим языком.
- Мелкие UX правки: кликабельные области, отступы, контраст, читабельность.

### Шаг 8. Подготовка к показу (видео 1–3 минуты)
- Сценарий демо:
  - запуск → shimmer → доска
  - перетаскивание между колонками
  - сортировка внутри колонки
  - показ “saving…” и успешного сохранения
  - демонстрация ошибки (можно выключить сеть) → откат/Retry
  - поиск/фильтр (если сделан)
- В видео акцентировать:
  - “минимум кликов”
  - “защита от ошибок”
  - “визуал KPI-DRIVE”

## Критерии готовности (Definition of Done)
- Доска строится из API ответа и корректно группирует по `parent_id`.
- Drag & Drop работает:
  - между колонками (меняет `parent_id`)
  - внутри колонки (меняет `order`)
- Сохранение на сервер работает и ошибки обрабатываются без “поломки” UI.
- UI выглядит современно: тема, типографика, состояния загрузки/ошибок, анимации.

## Примечания/риски (учтем заранее)
- API “save” в примере содержит **две пары** `field_name/field_value`. Если API реально ожидает по одному полю за запрос, сделаем два последовательных запроса с единым “saving” состоянием.
- Периоды в fetch и save в ТЗ различаются (2026-04 vs 2025-09). При реализации вынесем период в конфиг, чтобы легко переключать (и не ошибиться при демо).
